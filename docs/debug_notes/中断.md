# JZ2440 定时器中断调试记录 (Timer4 IRQ)

## 1. 实验背景
在 **JZ2440 Unified SDK** 项目中，实现 Timer4 中断实验。实验环境涉及 **NOR 直接启动** 和 **SPL 搬运到 SDRAM** 两种模式。

---

## 2. 关键故障点分析

### A. 变量失效（NOR 启动模式）
*   **现象**: 在 `TEXT_BASE = 0` (NOR 运行) 模式下，中断注册成功但跳转时 CPU 崩溃。
*   **原因**: NOR Flash 是只读的。当中断分发器 (`irq_handlers` 数组) 和全局计数器链接在地址 0 附近时，它们被分配到了 NOR Flash 空间。`hal_irq_register` 试图写入函数指针的操作被硬件静默忽略。
*   **解决**: 
    1. 基础实验推荐运行在 **SDRAM (0x30000000)** 中，以支持全局变量的动态读写。
    2. 即使在 SRAM 运行，也需确保 `.data` 和 `.bss` 段位于可写区域。

### B. 中断向量表冲突 (Vector Redirection)
*   **现象**: 程序搬运到 SDRAM 运行后，开启中断导致系统陷入无限重启（Reboot Loop）。
*   **原因**: 
    1. S3C2440 硬件中断发生时，无论主程序在何处，CPU 永远固定跳转到地址 **`0x18`** (IRQ 向量位置)。
    2. 在 NOR 启动模式下，`0x18` 属于 Bootloader (SPL) 的地盘。
    3. 如果 Bootloader 里的向量表没有正确处理中断，或者试图调用主程序中才有的 C 函数，会导致非法指令或取指异常。
*   **解决**: 实施 **“中断向量重定向”**。修改 Bootloader (`common/boot/nor_start.S`)，将其 IRQ 向量强制指向 SDRAM 对应位置：
    ```arm
    /* 在 Bootloader 的 0x18 地址处执行硬跳转 */
    ldr pc, =0x30000018
    ```

### C. 定时器内部状态清除 (TINT_CSTAT)
*   **现象**: 中断只触发一次，之后再无响应。
*   **原因**: S3C2440 的 Timer 4 (及其他定时器) 在定时器控制器内部有专门的使能和状态位。仅清除 INTC (中断控制器) 的标志位是不够的。
*   **细节**: 必须操作 `TINT_CSTAT` 寄存器。
    *   **[4:0]**: 中断使能 (1 = Enable)。
    *   **[9:5]**: 中断状态 (写 1 清零)。
*   **陷阱**: 使用 `TIMER->TINT_CSTAT = (1 << 9)` 这种简单的赋值会意外关闭所有使能位。
*   **标准写法**: 
    ```c
    /* 强制维持使能位并清除状态位 */
    TINT_CSTAT = 0x1F | (1 << (offset - 10 + 5)); 
    ```

## 3. 最终确认的架构方案

### 1. 启动汇编自适应 (`s3c2440_start.S`)
*   实现了独立的 **SVC 栈 (2KB)** 和 **IRQ 栈 (1KB)**，防止现场保护时损坏主程序上下文。
*   支持通过运行地址探测识别运行环境。

### 2. 中断分发流程
1.  **硬件层**: 产生中断 -> 跳到 `0x00000018` (NOR Flash)。
2.  **重定向层**: `ldr pc, =0x30000018` (转发到 SDRAM)。
3.  **现场保护层**: `stmdb sp!, {r0-r12, lr}` (保存到 SDRAM IRQ 栈)。
4.  **C 分发层**: 读取 `INTOFFSET` -> 调用回调 -> 清除 `TINT_CSTAT` -> 清除 `SRCPND/INTPND`。
5.  **恢复层**: `ldmia sp!, {r0-r12, pc}^` (恢复现场并返回)。

---

## 4. 调试结论
针对 JZ2440 的中断开发，**“直接赋值清除”** 和 **“显式向量转发”** 是确保系统稳定的两大支柱。目前 `05_timer_interrupt` 已验证在 500ms 周期下运行稳定，LED1 闪烁及串口 Ticks 计数正常。
