# 实验一：SRAM 调试环境搭建与流水灯实现

本实验是 JZ2440 裸机开发的第一步。目标是搭建完整的交叉编译与硬件调试环境，并通过控制 GPIO 实现最基础的 LED 流水灯逻辑。

---

## 🎯 实验目标
1.  搭建 `arm-none-eabi-gcc` 交叉编译环境。
2.  配置 OpenOCD 与 GDB，实现程序通过 JTAG 直接下载到 SoC 内部 SRAM 运行。
3.  掌握 S3C2440 GPIO 的寄存器配置方法（GPFCON/GPFDAT）。
4.  理解流水灯的硬件电路原理。

---

## 🛠️ 准备工作

### 1. 硬件需求
*   **开发板**：JZ2440 V3 (S3C2440A)。
*   **调试器**：OpenJTAG (或其它兼容 FT2232 的 JTAG 调试器)。
*   **连接**：使用 JTAG 排线连接调试器与板子的 JTAG 接口，并将板子设为 **NOR 启动**（仅为确保复位后 CPU 处于可控状态）。

### 2. 软件环境
*   **工具链**：`arm-none-eabi-gcc` (建议 13.2.1 或更高)。
*   **调试器**：`openocd` 与 `gdb-multiarch`。
*   **驱动**：确保系统识别 JTAG 调试器的 FTDI 芯片。

---

## 📚 硬件原理分析

参考 `docs/schematics/JZ2440_V3电路图.pdf`：
*   **LED 接口**：
    *   LED1 -> GPF4
    *   LED2 -> GPF5
    *   LED4 -> GPF6
*   **控制逻辑**：
    *   引脚输出 **低电平 (0)**：LED 点亮。
    *   引脚输出 **高电平 (1)**：LED 熄灭。
*   **上拉电阻**：板载已焊接 10K 外部上拉，无需额外配置 `GPFUP` 寄存器。

---

## 💻 软件实现

### 1. 启动代码 (`src/cpu/start.S`)
由于程序直接载入 SRAM 运行，我们需要最简化的初始化：
*   关闭看门狗（防止调试期间系统复位）。
*   设置栈指针（SRAM 空间 4KB，地址 0x00000000 - 0x00000FFF）。
*   跳转到 C 语言入口 `main`。

### 2. 核心逻辑 (`src/main.c`)
通过操作 `GPFCON` 寄存器将引脚设为“Output”模式，并在循环中修改 `GPFDAT` 的位状态：

```c
/* 设置 GPF4/5/6 为输出模式 */
GPFCON &= ~((3 << 8) | (3 << 10) | (3 << 12));
GPFCON |=  ((1 << 8) | (1 << 10) | (1 << 12));

/* 循环点亮 */
GPFDAT &= ~(1 << 4); // 点亮 LED1
delay(100000);
GPFDAT |= (1 << 4);  // 熄灭 LED1
```

## 🚀 编译、烧录与调试

### 步骤 1：编译
在根目录执行以下命令生成二进制文件：
```bash
make clean
make
```
这会生成 `jz2440.elf` (带调试信息) 和 `jz2440.bin` (裸机二进制)。

### 步骤 2：启动调试网关 (OpenOCD)
在一个独立的终端运行：
```bash
openocd -f scripts/jz2440.cfg
```
*注：若提示连接失败，请尝试在启动命令的同时按下板子上的硬件复位键。*

### 步骤 3：加载运行 (GDB)
在另一个终端运行：
```bash
gdb-multiarch jz2440.elf -x scripts/gdbinit
```
GDB 脚本会自动连接 OpenOCD、将程序下载到 SRAM 并停在 `_start` 处。输入 `c` (continue) 即可看到流水灯运行。

---

## ⚠️ 注意事项与技巧
1.  **SRAM 限制**：S3C2440 的内部起始 SRAM 只有 4KB。如果程序体积超过 4KB，必须进行 SDRAM 初始化并执行“代码重定位”，这将在后续实验中讲解。
2.  **JTAG 速率**：本分支配置的 `adapter speed` 为 200kHz，以确保在 CPU 未开启 PLL（运行在 12MHz）时的稳定性。
3.  **寄存器定义**：所有寄存器基地址均可在 `include/s3c2440_soc.h` 中找到，建议配合 **S3C2440A 数据手册** 阅读。
