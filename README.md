# 实验四：S3C2440 内存搬运性能深度测试 (CPU vs DMA)

本实验在 JZ2440 开发板上实现了多种模式下的内存搬运（Memcpy），并对比了在不同数据长度及 Cache 配置下，CPU 软件搬运与 DMA 硬件搬运的真实性能。

---

## 🎯 实验目标
1.  **多维度性能对比**：测试 4KB 到 1MB 数据在以下四种模式下的搬运速度：
    *   **CPU (None)**: 关闭所有 Cache 和 MMU（纯 SDRAM 访问）。
    *   **CPU (I-On)**: 仅开启指令缓存 (I-Cache)。
    *   **CPU (I+D)**: 同时开启指令和数据缓存 (D-Cache + MMU)。
    *   **DMA**: 使用硬件 DMA 控制器搬运（无 Cache 环境）。
2.  **验证一致性**：确保在开启 D-Cache 这种复杂环境下，DMA 搬运的数据依然能够通过 CPU 校验。

---

## 📊 典型测试结果 (吞吐量 Mbps)
在 HCLK=100MHz, FCLK=400MHz 配置下，各模式在不同数据长度下的表现：

| 搬运方法 | 4KB | 16KB | 64KB | 256KB | 1024KB | 备注 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **CPU (None)** | 8.19 | 8.11 | 8.10 | 8.09 | 8.09 | 受限于 SDRAM 等待周期。 |
| **CPU (I-On)** | 27.30 | 27.30 | 26.96 | 26.91 | 26.90 | 指令执行效率提升约 3 倍。 |
| **CPU (I+D)** | **409.60** | **327.68** | **327.68** | **319.68** | **316.79** | **最快**，利用了 D-Cache 突发读写。 |
| **DMA** | 81.92 | 74.47 | 74.47 | 73.84 | 73.73 | 纯硬件速度，性能极其稳定。 |

*注：CPU (I+D) 在小数据量（4KB）时表现极高，是因为数据可能完全命中 Cache 线，且循环开销极低。*

## 🛠️ 实验过程中遇到的问题与解决方案 (关键陷阱)

### 1. 计时器测量失效 (TCON 寄存器陷阱)
*   **原因**：S3C2440 的 `TCON` 寄存器中，Bit 22 是自动重载，Bit 21 是手动更新。初始化顺序错误导致计时器不计数。
*   **修正**：严格遵循 `Manual Update (On) -> Manual Update (Off) -> Start & Auto Reload (On)` 的顺序。

### 2. Cache 一致性问题 (DMA 校验失败)
*   **原因**：DMA 不经过 Cache。CPU 开启 D-Cache 时会读取 Cache 中的旧数据。
*   **修正**：DMA 搬运前执行 `Clean D-Cache` 刷回内存；搬运后执行 `Invalidate D-Cache` 强制 CPU 从内存读取真值。

### 3. ARM920T 特有的 Cache 几何结构
*   **原因**：ARM920T 的 D-Cache 是 **16KB, 64-way, 8 sets (segments)**。
*   **解决**：必须通过 64x8 的双重循环，使用特定的 `index = (way << 26) | (set << 5)` 格式来清理 Cache。

### 4. 栈指针 (SP) 溢出与回绕
*   **现象**：程序打印标签损坏或乱码。
*   **原因**：SP 设在 `0x34000000` 超出了 64MB SDRAM 范围导致回绕。
*   **修正**：将 SP 下移至 `0x33F00000` 确保安全。

### 5. 64 位算术运算导致的栈破坏
*   **现象**：使用 `unsigned long long` 计算 Mbps 后，输出变为 `0 KB`。
*   **原因**：在裸机环境下，64 位除法会调用庞大的编译器辅助函数，在栈空间受限时极易导致栈溢出。
*   **解决**：将测试参数数组移至全局数据段，并简化计算逻辑为 32 位安全运算。

## 💡 深度探讨：既然 CPU (I+D) 更快，为什么还要 DMA？

在本实验中，开启全 Cache 的 CPU 搬运速度约为 DMA 的 **4 倍**。但在嵌入式系统中，DMA 依然不可或缺：

1.  **释放 CPU 算力**：CPU 搬运时必须全力以赴，无法处理其他任务；DMA 启动后，CPU 可以并发处理数学运算、协议解析等复杂任务，显著提升**系统总吞吐量**。
2.  **避免缓存污染 (Cache Thrashing)**：CPU 搬运大数据量（如 1MB）会强行刷掉 D-Cache 中原本存放的“热点”业务数据。搬运结束后，CPU 执行正常代码会产生大量 Cache Miss。DMA 则不破坏 Cache 状态。
3.  **外设通信的桥梁**：DMA 真正的战场是内存与外设（网卡、摄像头、音频）之间的异步交互。若无 DMA，CPU 必须频繁响应中断去读取 I/O 寄存器，会导致系统实时性崩溃。
4.  **确定性耗时**：CPU 耗时受 Cache 命中率和中断影响有波动，而 DMA 拥有独立总线控制权，搬运耗时极其固定，适合 LCD 刷新等实时应用。

## 💻 核心配置代码摘要 (DMA)
```c
void dma_memcpy(unsigned int word_count) {
    DMASKTRIG3 = 0;
    DISRC3 = SRC_ADDR; DIDST3 = DST_ADDR;
    /* [30]HCLK, [28]Burst4, [27]Whole, [23]SW, [22]NoReload, [21:20]Word */
    DCON3 = (0<<31)|(1<<30)|(1<<28)|(1<<27)|(0<<23)|(1<<22)|(2<<20)|word_count;
    DMASKTRIG3 = (1<<1) | (1<<0); 
    while ((DSTAT3 & (1<<20)));   
}
```

## 📝 实验总结
1.  **Cache 是性能核心**：在 S3C2440 上，开启 D-Cache 后的 CPU 拷贝速度远高于 DMA。这是因为 CPU 能够利用 L1 Cache 的高带宽和突发传输。
2.  **DMA 是系统加速器**：DMA 的价值不在于单次拷贝的绝对速度，而在于其异步处理能力和对 CPU 资源的释放。在高性能嵌入式设计中，合理分配 Cache 处理短促任务与 DMA 处理背景流数据，是系统优化的关键。
